# Phase 3: Evidence-Based Proctoring

Status: planning complete, implementation not started.

## Purpose

Phase 3 adds lightweight, evidence-based proctoring to the candidate workspace.
The goal is not to prevent cheating — browsers cannot do that reliably — but to
give employers a clear, honest picture of candidate behavior during the assessment
so they can decide how much weight to put on the result.

Phase 3 does not add:
- Video streaming or continuous recording
- Paste blocking
- Dev tools detection (unreliable, produces false positives)
- Mandatory webcam (opt-in only)

## Design philosophy

Every proctoring signal is logged and surfaced to the employer, never silently discarded
or used to auto-reject a candidate. The employer sees the evidence and decides.

Honest-candidate-friendly: brief focus-loss events (system notification, window resize)
generate low-weight signals, not automatic flags.

## What Phase 3 delivers

### 1. Browser event logging

Track and persist:
- **Fullscreen exit** — candidate leaves fullscreen mode (logged, not blocked)
- **Focus loss / window blur** — candidate switches to another app or tab
- **Focus return** — paired with focus loss to measure duration away

Full-screen is requested on workspace load. If declined, the candidate proceeds
without fullscreen and that is noted. Exiting fullscreen is always logged.

Source: `document.addEventListener("fullscreenchange")` and
`document.addEventListener("visibilitychange")` / `window.addEventListener("blur")`.

### 2. Webcam consent and periodic snapshots

Optional. Candidate is asked for camera permission at assessment start with explicit
consent copy. If denied, the assessment proceeds normally — no penalty.

If granted:
- JPEG snapshot captured every 5 minutes (configurable)
- Additional snapshot on each focus-loss event > 30 seconds
- Final snapshot at submission
- Each snapshot uploaded directly to S3 via presigned URL from the API
- S3 path: `snapshots/{attempt_id}/{unix_timestamp}.jpg`
- Thumbnails shown in employer report with timestamps

No audio is recorded. No face-analysis or emotion detection.

### 3. Unified integrity score

Replaces the separate `ai_integrity_risk` field with a combined `integrity_score`
that draws from both AI collaboration signals and proctoring signals.

All thresholds are stored as named configuration values — not hardcoded — so they
can be tuned after seeing real pilot data.

See `03-unified-integrity-score.md` for rubric detail.

### 4. Employer report: Proctoring Signals section

New section in the evidence report, positioned after process evidence and before
submission review. Shows:
- Consent status (webcam granted / declined)
- Focus-loss event count and total duration
- Fullscreen exits
- Large paste events (carried over from Phase 2)
- Snapshot thumbnail strip (if webcam consented)
- Combined integrity flag with contributing signals

## Non-goals

- Preventing candidates from opening other apps or browser tabs
- Detecting or blocking OS-level window switching (Alt+Tab / Cmd+Tab)
- Audio recording
- Mandatory webcam (always optional)
- Video recording or streaming (post-Phase-3)
- Screen recording

## Deployment notes

- S3 bucket: the operator-configured ECS runner bucket (`SIGNALLOOP_RUN_BUCKET`), using the
  `snapshots/` prefix.
- Render already has `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_REGION`, and `S3_BUCKET`
  environment variables set. No new infrastructure needed.
- `EXECUTION_BACKEND=direct` (current Render config) does not affect snapshot upload — presigned
  URLs are generated by the API directly, independent of execution backend.

## Optional future enhancement (post-Phase-3)

- Client-side face-presence detection using a small on-device model (no server-side face analysis)
- Periodic active-attention prompts ("still there?" click)
- Assessment variant randomization to reduce answer sharing across candidates
