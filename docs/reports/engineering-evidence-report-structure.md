# Engineering Evidence Report Structure

## Sections

1. Metadata (attempt, candidate, assessment, timing, evaluator feedback mode)
2. Executive summary (narrative + evidence limits)
3. Overall recommendation (strong_advance / advance_with_followups / needs_review / do_not_advance)
4. Score breakdown (all categories with points, max, evidence string)
5. Rubric weights (pack-specific)
6. Public test results (last run summary, run count, initially failing tests)
7. Hidden test results (summary, seeded issue areas)
8. Feature/design implementation (score detail for that category)
9. FAVO interpretation (Frame / Ask / Verify / Own — label + evidence per dimension)
10. Candidate-written tests (files added/modified, function count, assertion count)
11. AI collaboration (prompt count, policy redirect count, flagged prompts, paste detection)
12. AI integrity risk (label: low/medium/high/critical, signals, score impact note)
13. Submission review (parsed candidate notes: what changed, tradeoffs, additional notes)
14. Process evidence (snapshot count, test run count, run timings)
15. Follow-up questions (generated from score evidence and flags)
16. Timeline (chronological event log)
17. Submitted code (file tree of the candidate's final snapshot, displayed as a tab-based file viewer in the employer UI)

## Source mapping

| Report section | Data sources |
|---|---|
| Score breakdown | Public test runs, hidden test runs, candidate verification run, AI interaction logs |
| Public test results | `TestRun` records with `run_type="public"` |
| Hidden test results | `TestRun` records with `run_type="hidden"`, submitted by backend |
| Feature/design implementation | Named tests in `feature_design_tests` pack config, hidden test run failures only |
| FAVO | Feature score, AI prompt count, test/run evidence, submission review answer count |
| Candidate-written tests | `TestRun` records with `run_type="candidate_verification"` — proving-test count |
| AI collaboration | `AIInteraction` records, pasted-code detection, large paste detection |
| AI integrity risk | Policy redirect counts, paste events, submission review completeness |
| Submission review | `final_explanation` and `decision_log` from `FinalSubmission` |
| Timeline | `AssessmentAttempt`, `CodeSnapshot`, `TestRun`, `AIInteraction` timestamps |
| Follow-up questions | Generated from hidden failures, AI flags, paste events, submission review gaps |
| Submitted code | Final `CodeSnapshot` files, sorted: `task_api/` first, `tests/` second, config last |

## Implementation notes

Reports are generated by `build_report()` in `apps/api/signalloop_api/reports.py` and
persisted as `EvidenceReport` records. Scores are deterministic estimates from captured
evidence. Manual evaluator review is required before hiring decisions.

The `llm_assisted_review` field is present in the JSON payload (`status: not_run`) but
is not displayed in the employer report UI until a bounded prompt and safety boundary
are added.
